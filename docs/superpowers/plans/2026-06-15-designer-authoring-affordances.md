# Designer Authoring Affordances Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Repair the authoring-UX regression from the 024 band-model reification so a blank report can be built into a grouped, master/detail invoice using the designer UI alone — by wiring the already-existing scope/group commands to the UI, creating lists data-first from the Data Source panel, and surfacing the list binding on the detail band where authors look.

**Architecture:** UI-only changes under `packages/jet_print/lib/src/designer/`. The domain model, serialization codec, schema version (=2), and render engine are untouched — so render goldens stay byte-identical. Two new compound controller methods bundle multi-node creation into one undo step via the existing `DefinitionEditCommand`; everything else is widget wiring that calls existing controller methods (`createListWithBand`, `createGroupWithHeader`, `setScopeCollection`, `createBoundElement`). New user-facing strings are added to the en/tr/de `.arb` files (regenerated by `flutter pub get`).

**Tech Stack:** Dart ≥3.6 / Flutter, `shadcn_ui` (designer widgets), `flutter gen-l10n` (.arb localization), `flutter_test` widget + golden tests. Repo is a pub workspace monorepo; the library is `packages/jet_print`.

## Refinements discovered during research (deviations from the design doc)

1. **No domain change.** The "nested scope is missing its collectionField" diagnostic already exists as invariant **I6** in `report_validation.dart`. The design's "single additive diagnostic in `report_validation.dart`" is therefore **not needed**. This plan touches **only** `lib/src/designer/` and the `.arb` files.
2. **Scope-node picker already done.** `_scopeInspector` already renders a schema-aware `_BindingField` (the design's "upgrade scope inspector" is already shipped). Only the **detail-band** inspector needs the new List section (Task 4).
3. **Validation surfacing is scoped to inline warnings.** `Diagnostic` carries only an optional `elementId` (no scope/band/group target), so generic per-node ⚠ badges and a top-bar summary cannot be wired without enriching the domain `Diagnostic` type or top-bar surgery. This plan surfaces the **critical unbound-list case inline** in the band/scope inspector (fully model-derived, Task 4). Per-node badges + a global summary are listed under **Deferred within this plan**.

```dart
  /// Creates a nested list (scope) iterating [collectionField] under
  /// [parentScopeId], pre-populated with one empty detail band, and selects that
  /// band — as ONE undoable step. The data-first entry point used by a Data
  /// Source collection drop/＋ and the Outline "Add list" action to build a
  /// master/detail. A no-op for an unknown parent scope.
  void createListWithBand(String parentScopeId, {String? collectionField}) {
    if (findScope(_document.definition, parentScopeId) == null) return;
    final Band band = Band(
        id: _ids.next('band'),
        type: BandType.detail,
        height: _defaultBandHeight(BandType.detail));
    final DetailScope scope = DetailScope(
      id: _ids.next('scope'),
      collectionField: collectionField,
      children: <ScopeNode>[BandNode(band)],
    );
    _commit(DefinitionEditCommand(
      label: 'Add list',
      transform: (ReportDefinition d) =>
          addScopeChild(d, parentScopeId, NestedScope(scope)),
      selection: Selection.band(band.id),
    ));
  }
```

All referenced symbols (`findScope`, `Band`, `BandType`, `_ids`, `_defaultBandHeight`, `DetailScope`, `ScopeNode`, `BandNode`, `NestedScope`, `DefinitionEditCommand`, `addScopeChild`, `Selection`, `ReportDefinition`) are already imported/used by the neighbouring `createScope`/`addDetailBand` methods — no new imports.

  /// Creates a group level on scope [scopeId] together with its header band, and
  /// selects the header band so its key + flags are edited right there (the
  /// edit-the-group-from-its-band pattern) — as ONE undoable step. The group
  /// gets a fresh unique id/name and a constant placeholder key the author then
  /// replaces. A no-op for an unknown scope.
  void createGroupWithHeader(String scopeId) {
    if (findScope(_document.definition, scopeId) == null) return;
    final String groupId = _ids.next('group');
    final Band header = Band(
        id: _ids.next('band'),
        type: BandType.groupHeader,
        height: _defaultBandHeight(BandType.groupHeader));
    final GroupLevel group = GroupLevel(
      id: groupId,
      name: groupId, // unique by construction; a display label only
      key: '0', // a constant placeholder key (parses); the author edits it
      header: header,
    );
    _commit(DefinitionEditCommand(
      label: 'Add group',
      transform: (ReportDefinition d) => addGroup(d, scopeId, group),
      selection: Selection.band(header.id),
    ));
  }

  /// The "List" section on a DETAIL band (Surface A): the collection its
  /// enclosing scope iterates, editable where the author looks. A root-scope
  /// detail band shows a read-only "main dataset" label (the root iterates the
  /// records themselves, no collection field); a nested-list detail band shows
  /// the same schema-aware binding picker the scope inspector uses, plus an
  /// inline warning when the list is unbound.
  List<Widget> _bandListSection(
    JetReportDesignerController controller,
    String bandId,
    ShadThemeData theme,
    JetPrintLocalizations l10n,
    JetDataSchema? schema,
  ) {
    final DetailScope? scope = findScopeOfBand(controller.definition, bandId);
    if (scope == null) return const <Widget>[];
    final bool isRoot = controller.definition.body.root.id == scope.id;
    if (isRoot) {
      return <Widget>[
        SectionLabel(l10n.propertiesList),
        const SizedBox(height: 8),
        Text(l10n.propertiesListRootSource, style: theme.textTheme.muted),
      ];
    }
    return <Widget>[
      SectionLabel(l10n.propertiesList),
      const SizedBox(height: 8),
      _BindingField(
        fieldKey: const ValueKey<String>('$_p.field.bandCollection'),
        value: scope.collectionField ?? '',
        placeholder: l10n.bindingCollectionHint,
        clearTooltip: l10n.bindingClearTooltip,
        fields: _scopeCollectionChoices(schema, controller, scope.id),
        pickerTooltip: l10n.bindingFieldPickerTooltip,
        pickerKeyPrefix: '$_p.field.bandCollection.pick',
        onSet: (String v) => controller.setScopeCollection(scope.id, v),
        onClear: () => controller.setScopeCollection(scope.id, null),

  final FieldDef field;

  @override
  Widget build(BuildContext context) {

            behavior: HitTestBehavior.opaque,
            onTap: () {
              final JetReportDesignerController c =
                  DesignerScope.of(context, listen: false);
              c.createListWithBand(

                collectionField: field.name,

/// The scope a new list for a collection should nest under: the scope already
/// bound to [parentCollection] (its parent in the schema tree), or the root
/// scope for a top-level collection / when no bound parent scope exists yet.
String _resolveParentScope(ReportDefinition def, String? parentCollection) {
  if (parentCollection == null) return def.body.root.id;
  String? found;
  void walk(DetailScope s) {
    found ??= s.collectionField == parentCollection ? s.id : null;
    for (final ScopeNode n in s.children) {
      if (n is NestedScope) walk(n.scope);
    }
  }

  walk(def.body.root);
  return found ?? def.body.root.id;
}

void _handleFieldDrop(
  FieldDragData data,
  Offset globalOffset,
  JetReportDesignerController controller,
  CanvasViewTransform transform,
  DesignTimeLayout layout,
) {
  final RenderObject? object = _pageKey.currentContext?.findRenderObject();
  if (object is! RenderBox) return;
  final Offset local = object.globalToLocal(globalOffset);
  final JetOffset page =
      JetOffset(local.dx / transform.scale, local.dy / transform.scale);
  final String? bandId = layout.bandIdNear(page);
  if (bandId == null) return;
  if (data.isCollection) {
    // A collection nests under the scope that owns the drop band (furniture /
    // once-bands resolve to the root master scope).
    final DetailScope? enclosing =
        findScopeOfBand(controller.definition, bandId);
    controller.createListWithBand(
      enclosing?.id ?? controller.definition.body.root.id,
      collectionField: data.fieldName,
    );
    return;
  }
  controller.createBoundElement(
    bandId: bandId,
    at: layout.toBandLocal(bandId, page),
    expression: '\$F{${data.fieldName}}',
  );
}

## Deferred within this plan (explicit, not silent)

These were in the design's comprehensive intent but are intentionally **out of this plan** to keep it domain-free and free of speculative code. Each is a clean follow-up:

1. **Per-node ⚠ badges in the Outline + a global diagnostics summary.** Blocked cleanly: the domain `Diagnostic` type carries only an optional `elementId`, so a diagnostic cannot be mapped to a scope/group/band node generically. Doing this well wants a small domain change — add a target id to `Diagnostic` (and have `validate()` populate it). That is a separate, schema-version-neutral domain task. *This plan instead surfaces the critical unbound-list case inline (Task 4), where the author is already looking.*
