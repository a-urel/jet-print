# Epic E5 — Mobile / Touch Support — Findings

**Status:** implementation complete on branch `e5-mobile-support` (not merged,
not pushed); manual sim/emulator smoke (SC-E5-002) **pending user confirmation**.
**Spec:** [2026-06-21-e5-mobile-support-design.md](./2026-06-21-e5-mobile-support-design.md)
**Plan:** [../plans/2026-06-21-e5-mobile-support.md](../plans/2026-06-21-e5-mobile-support.md)

Mirrors the E4 findings record. Like E4, E5 is **verify + harden + targeted
affordances**, validated locally — here on the iOS Simulator + an Android
emulator (Xcode 26.5, Android SDK 36 confirmed on the dev Mac).

## 1. Build gate (Task 1)

The de-risk gate — build a real mobile consumer **before** any hardening —
**passed on both platforms**:

- `flutter build apk --debug` → built `app-debug.apk` (~585 s first run, incl.
  Gradle + Android NDK install).
- `flutter build ios --debug --no-codesign` → built `Runner.app` (~132 s; pod
  install ~0.9 s).
- `flutter analyze` (repo root, strict workspace config) → `No issues found!`

Generated `ios/` + `android/` runner dirs and relaxed the playground's platform
guard to admit iOS + Android. No hidden transitive Dart/plugin blocker surfaced —
the verify+harden framing held.

**Two build-time observations (deferred, not blockers):**
- **`printing` has no Swift Package Manager support yet.** The iOS build works
  via CocoaPods, but Flutter warns SPM may become mandatory; `printing` 5.15.0
  exists. Revisit in the E6/E7 dependency review.
- **Mobile bundle IDs are `com.example.*`** (the `flutter create` default; the
  repo's existing runners are already inconsistent — macOS `com.jetprint.*`,
  Linux `com.example.*`). Bundle-ID / org normalization is **packaging work,
  explicitly E7 scope** — deferred deliberately, not an oversight.

## 2. Output harden (Task 2)

The automated render + PDF + PNG export test under
`debugDefaultTargetPlatformOverride = TargetPlatform.android` / `iOS`
(`mobile_render_export_test.dart`) **passed on the first run** — the export path
carries no platform-conditional Dart bug (unlike E4, where the browser surfaced
a real `double.toString` divergence; mobile uses the Dart VM/AOT runtime, so the
number formatting E4 normalized stays correct). **No library hardening was
required.**

This VM test guards platform-conditional Dart code; the real Impeller/Skia
device rendering is verified by the §6 manual smoke.

## 3. Print seam (Task 3)

The print seam (`Printing.info()` + `Printing.layoutPdf`) routes to the **native
iOS/Android print/share sheet**; its mobile semantics are now documented at the
seam (no desktop dialog; cancel-as-success best-effort, as on web;
`PrintUnavailableException` still covers genuinely unsupported environments). The
printer suite stays green (8/8). **No iOS `Info.plist` / Android manifest entry
was required** — the `printing` plugin's build passed with the default runner
config.

## 4. Touch affordances (Tasks 5–7)

- **Active-pointer-kind signal** — the canvas tracks the last pointer-down kind
  and exposes `_isTouch`, keyed on the pointer kind (not `Platform.isX`), so a
  touchscreen laptop and a keyboard-tablet both stay correct.
- **Touch-sized hit areas** — resize handles + band divider grow to a 44pt hit
  target and the scrollbars to 20px **when touch is active**; the drawn handle
  is unchanged. **Goldens are byte-identical** (the touch branch never executes
  under goldens, which simulate no touch pointer).
- **Long-press = right-click** — `longPressEnabled: true` on the canvas context
  menu opens it on long-press; `onTapDown` already selects the pressed element,
  so the menu acts on it. (Note: the existing inventory claim that Undo/Redo were
  keyboard-only was wrong — the top bar already carries Undo/Redo, Cut/Copy/Paste,
  and zoom, so no toolbar code was needed; Delete/Duplicate reach touch via this
  long-press menu.)

## 5. Phone-width layout (Task 8)

`_minShellWidth` lowered 600 → 360 so a ~390pt phone authors in the real narrow
(rail) layout — 48px toolbox + surface + 48px collapsed right-rail — instead of a
600px horizontally-scrolling shell. No residual cramping beyond the documented
limitation below was observed in the widget tests.

## 6. Manual smoke (SC-E5-002) — PENDING USER CONFIRMATION

To be run by the user on the **iOS Simulator** + an **Android emulator**
(`flutter run` the playground on each). Checklist:

- [ ] Every playground sample renders correctly (invoice, label, barcode, menu,
      nested-list, packing-slip, payroll).
- [ ] PDF export + PNG export work (the OS share sheet appears).
- [ ] Print opens the native print/share sheet.
- [ ] Authoring: tap-select an element; **long-press** opens the context menu;
      **resize** via the enlarged (finger-grabbable) handle; **move** by drag;
      **zoom** via the +/−/fit buttons.
- [ ] Phone-width layout: the designer authors without a horizontal-scroll shell;
      the right panel reaches via its rail/overlay.

**Round 1 (iOS Simulator, user-run) — caught 3 issues, all fixed:**
1. **No safe area** — the designer sat under the notch / Dynamic Island / home
   indicator. Fixed: the playground shell is wrapped in `SafeArea` (`f88f46a`).
2. **Scroll didn't work on touch** — the canvas disables drag-to-scroll so its
   gestures win, leaving only the finger-unfriendly scrollbar. Fixed: on touch a
   one-finger drag on empty canvas pans the viewport (marquee stays the mouse
   gesture) (`2b9eccb`).
3. **Resize handles opened the context menu instead of resizing** —
   `ShadContextMenuRegion` defaults `tapEnabled=true` on iOS/Android, so a plain
   touch-down on a handle showed the menu. Fixed: `tapEnabled:false` (menu via
   long-press/right-click only) plus a long-press-drag resize on the handles
   (`2b9eccb`). New regression tests cover both 2 and 3.

**Round 2 (iOS Simulator, user-run) — caught 3 more issues, all fixed:**
1. **Only the first demo was reachable** on a phone — the playground's
   scrollable demo strip and the app-global theme/locale toggle cluster share a
   `Stack`, and at phone width the cluster overlaid (and hid) the strip's
   right-hand tabs. Fixed: below 600px the shell stacks a compact toggle row
   ABOVE the full-width demo strip, so every demo is reachable by swiping
   (`0b07f54`).
2. **Designer|Preview switch too wide on a phone** — it now drops its segment
   labels and renders icon-only below `kBarVeryNarrowWidth` (600px of bar
   width), keeping the glyphs + a hover tooltip / accessible label. Scoped to
   the designer so the report-preview goldens stay byte-identical (`48afd12`).
3. **Editable zoom % field crowds a phone bar** — hidden below the same width;
   the zoom-out / zoom-in buttons remain (`48afd12`). New designer-bar tests at
   390px cover both 2 and 3; the demo-strip fix has a phone-width playground
   test.

**Round 3 (iOS Simulator, user-run) — preview parity:**
- **Android emulator "not found"** in the VS Code debugger — same booted-vs-cold
  mismatch as iOS: a running emulator reports id `emulator-5554` / name "sdk
  gphone64 …", neither containing its AVD name, so the launch config's
  `Pixel_9_Pro` only matches a *cold* AVD. Fixed the (gitignored, local) launch
  config to `deviceId: "emulator"`.
- **Preview must match the designer.** Round 2 scoped the icon-only mode switch
  + zoom-field hide to the designer to keep the report-preview goldens frozen;
  the user asked the preview to match. The preview's Designer|Preview switch now
  goes icon-only and its editable zoom % field hides below `kBarVeryNarrowWidth`
  (the +/- buttons stay). This **intentionally regenerates** five report-preview
  goldens (rendered_invoice light/dark/page2, label_sheet, formatted_value) —
  the change is confined to the 52px top bar; each golden's rendered PAGE
  content is unchanged (verified by inspection). A deliberate, user-directed
  exception to the otherwise-frozen goldens.

**Result:** _Round 4 re-smoke pending (iOS + Android) — verify the preview parity
+ the Android leg, and the remaining checklist; record PASS/FAIL per platform._

## 7. Known limitations (accepted)

- **Phone authoring is "usable," not "optimal."** Tablet is the primary
  authoring target (spec §7); a ~390pt canvas is inherently tight. Residual
  cramping is accepted per the lean Phase-3 decision rather than expanded.
- **Mobile bundle IDs / org** (`com.example.*`) and the **`printing` SPM** gap
  are E6/E7 packaging-and-dependency concerns (§1), not E5.
- **Mobile file UX is minimal** — save/export use the OS share sheet; a richer
  open/save flow is E7.
