# E2 — Resilience & Stress: Findings

- **Date:** 2026-06-20
- **Spec:** [2026-06-20-e2-resilience-stress-design.md](./2026-06-20-e2-resilience-stress-design.md)

## Committed stress test

`test/rendering/resilience/stress_dirty_dataset_test.dart` renders a flat report
over N = 50,000 rows with a wrong-type `amount` every 100th row. It
asserts resilience invariants only (no crash; per-row data warnings ≤
`DiagnosticBudget.kMaxPerRowDataDiagnostics`; a suppression summary present;
clean rows sum correctly). Time and RSS are logged, not gated.

N = 50,000 was chosen as the committed value: 503 ms wall time and 74 MB RSS
delta on this machine — well within a comfortable CI budget. At N = 50,000 there
are 500 dirty rows, so 400 are suppressed and the suppression-summary invariant
holds cleanly.

## Escalating-N exploration (one-off; not committed)

Exploration was run by temporarily patching `_n` in the test file; results below
were measured on an Apple M-series machine (macOS 25.5.0, Dart/Flutter SDK,
single-process `flutter test`). All runs PASSED their invariants except N =
10,000 (see note).

| N | pages | fill+layout wall time | RSS Δ | outcome |
|---|------:|----------------------:|------:|---------|
| 10,000 | 209 | 201 ms | 28 MB | QUIRK — suppression invariant inapplicable (see note) |
| 50,000 | 1,042 | 503 ms | 74 MB | OK (committed N) |
| 100,000 | 2,084 | 813 ms | 132 MB | OK |
| 250,000 | 5,209 | 1,749 ms | 319 MB | OK |
| 500,000 | 10,417 | 3,241 ms | 631 MB | OK |
| 1,000,000 | 20,834 | 6,382 ms | 1,242 MB | OK (slow but stable) |

**N = 10,000 note:** with `_dirtyEvery = 100` the dirty count is exactly
`ceil(10000/100) = 100`, which equals `kMaxPerRowDataDiagnostics`. No rows are
suppressed, so the suppression-summary info entry is absent. This is correct
engine behaviour at the exact cap boundary, not a bug; the committed N = 50,000
(500 dirty rows) keeps the test well into the suppression region.

Observed breaking point: none observed up to N = 1,000,000. Memory and time grew
strictly linearly with N (≈ 1.24 MB / 1,000 rows; ≈ 6.4 µs / row). No OOM, no
crash, diagnostic list remained bounded at 101 entries for all Ns.

## Recommendation (E2b gate)

The fill stage materializes every band for every row into one `List<FilledBand>`
before returning (`report_filler.dart`), and PDF export assembles the whole
document in memory. Based on the table above:

Growth is strictly linear with no cliff: 1 M rows consumed 1.24 GB process RSS
(including the Flutter VM baseline) and completed in 6.4 s. At the practical
embed volumes for jet_print (single-device, local data, reports up to low-tens-
of-thousands of rows), memory and time stay entirely comfortable. The 50 k
committed stress point (74 MB Δ, 503 ms) is the target ceiling for the library's
stated embed use-case.

Final call: **E2b (streaming fill / streaming export) is NOT required for the
embed gate; defer indefinitely.** The current in-memory materialize-all
architecture is safe at all realistic embed volumes. Should a future user surface
a multi-hundred-thousand-row report that is unacceptably slow on a constrained
device, this finding table (specifically the N = 500,000 → 631 MB / 3.2 s and N
= 1,000,000 → 1,242 MB / 6.4 s rows) provides the motivating evidence to open
E2b at that time.
